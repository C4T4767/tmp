# 기능명세

## 서비스 목표

해외직구 건강기능식품 또는 관련 상품의 성분을 분석하고, 국내 반입차단 여부와 사용자/그룹 건강정보 기준 주의 여부를 확인한다.

## 관련 기술스택

서비스 구현에 사용할 기술스택은 `docs/tech-stack.md`에 정리한다.
현재 레포에 적용된 웹 프로토타입 스택과 실제 서비스 도입 예정 스택을 분리해서 관리한다.

## AI 활용 범위

이 서비스의 AI 핵심 가치는 단순 검색이 아니라 쇼핑몰 상품 정보를 안전 분석에 쓸 수 있는 형태로 바꾸고, 사용자 건강정보와 연결해 이해하기 쉬운 주의 결과를 만드는 것이다.

### AI가 담당하는 일

- 쇼핑몰 상품 페이지에서 상품명, 브랜드, 용량, 수량, 성분표, 주의사항을 추출한다.
- 쇼핑몰마다 다른 상품명을 비교해 같은 상품인지 판단한다.
- 상품 이미지와 옵션 정보를 함께 보고 같은 상품 후보를 걸러낸다.
- `melatoni`, `melatonin`, `멜라토닌`처럼 다르게 적힌 성분명을 표준 성분으로 매칭한다.
- 공공데이터 기능성 문구를 `면역`, `항산화`, `장 건강` 같은 표준 목적 태그로 정규화한다.
- 공공데이터의 긴 주의사항 문장을 성분별 최종 주의 근거로 요약한다.
- 사용자 온보딩 정보와 성분 주의사항을 비교해 왜 주의가 필요한지 설명 문구를 생성한다.
- 사용자가 추가로 입력한 건강 메모를 성분 주의사항과 의미 기반으로 비교한다.
- 판매처별 댓글을 긍정, 부정, 기타로 분류하고 배송, 가격, 품질, 포장 같은 관점별로 요약한다.

### AI를 쓰지 않는 일

- 반입차단 성분 여부는 공공데이터 매칭 결과를 우선한다.
- 임신, 고혈압, 고지혈증, 당뇨처럼 명확한 온보딩 값은 우선 키워드 규칙으로 판단한다.
- AI 판단만으로 구매 금지 결정을 내리지 않는다.
- AI 결과는 `product_ingredient_analyses`, `user_ingredient_cautions`에 저장해 재사용한다.

### MVP 판단 방식

MVP는 규칙 기반 판정과 AI 보조 판정을 같이 사용한다.

1. 공공데이터 반입차단 성분은 DB 매칭으로 즉시 `BLOCKED` 처리한다.
2. 온보딩 조건은 조건별 키워드로 먼저 매칭한다.
3. 키워드로 애매한 경우에 LLM 또는 pgvector 기반 의미 비교를 사용한다.
4. 최종 사용자 문구는 AI가 생성하되, 근거는 성분명, 공공데이터 주의사항, 온보딩 조건에서 가져온다.

## 홈

### 실시간 검색 TOP 10

- 홈 진입 시 실시간 검색 TOP 10 상품을 보여준다.
- TOP 10 랭킹은 Redis Sorted Set `search:rank:realtime`을 사용한다.
- Redis에는 `product_id`만 저장한다.
- 상품명, 이미지, 상태는 PostgreSQL `products`에서 조회한다.
- 상품을 클릭하면 상품 상세 화면으로 이동한다.

### 구매 여부 확인

- 홈 진입 시 아직 답하지 않은 구매 여부 확인 항목을 조회한다.
- 확인 대상이 있으면 홈에서 “구매하셨나요?” UI를 보여준다.
- 확인 대상이 여러 개이면 모달 안에서 옆으로 넘기고 점 표시로 현재 항목을 보여준다.
- 사용자가 모달을 매번 보기 싫을 수 있으므로 홈 왼쪽 위에 구매확인 토글을 제공한다.
- 구매확인 토글 상태는 쿠키에 저장한다.
- 토글이 꺼져 있으면 확인 대상이 있어도 자동 모달을 띄우지 않는다.
- 사용자가 구매했다고 답하면 구매 기록으로 저장한다.
- 사용자가 구매하지 않았다고 답하면 확인 항목을 더 이상 표시하지 않는다.
- 오래된 확인 항목은 만료 처리한다.

## 상품 링크 분석

### 입력

- 사용자는 해외 쇼핑몰 상품 URL을 붙여넣는다.
- 사용자는 다른 쇼핑몰 앱이나 브라우저에서 공유하기 버튼을 눌러 SafeBuy로 상품 URL을 보낼 수 있다.
- 지원 쇼핑몰 예시는 아마존, 이베이, 아이허브, 비타트라, 오플닷컴, 쿠팡, 네이버 가격비교, 11번가이다.

### 분석 진행 방식

- 링크 분석은 job 기반으로 처리한다.
- 모든 링크 분석 요청은 `product_import_jobs`에 저장한다.
- 앱 안에서 URL을 붙여넣은 경우 `input_type = URL_PASTE`로 저장한다.
- 다른 앱의 공유하기로 들어온 경우 `input_type = APP_SHARE`로 저장한다.
- 클라이언트는 분석 job을 생성한 뒤 SSE로 진행 상태를 구독한다.
- 화면에는 개발자용 이벤트명이 아니라 사용자가 이해할 수 있는 단계명을 보여준다.

### 다른 앱에서 공유하기

- 사용자가 쇼핑몰 앱 또는 브라우저에서 공유하기를 누르고 SafeBuy를 선택한다.
- SafeBuy 앱은 공유받은 URL로 분석 job을 생성한다.
- 분석 진행 화면으로 이동해 SSE 진행 상태를 보여준다.
- 분석이 완료되면 생성 또는 매칭된 상품 상세 화면으로 이동한다.
- 이미 분석 완료된 URL이면 새 분석을 생략하고 기존 상품 상세 화면으로 바로 이동할 수 있다.

### SSE 단계

- STORE_DETECTED: 쇼핑몰 확인
- SOURCE_CRAWLED: 원본 상품 정보 수집
- INGREDIENTS_EXTRACTED: 성분 추출
- SAFETY_ANALYZED: 공공데이터 안전 분석
- PRICE_CANDIDATES_SEARCHED: 다른 쇼핑몰 가격 후보 검색
- OFFERS_MATCHED: 같은 상품 여부 검증
- REVIEWS_SUMMARIZED: 판매처별 댓글 요약
- COMPLETED: 분석 완료
- FAILED: 분석 실패

### 원본 URL 처리

- 사용자가 붙여넣은 URL의 판매처는 원본 판매처로 본다.
- 원본 URL에서 수집한 가격은 같은 상품 검증 없이 `product_offers`에 확정 저장할 수 있다.
- 단, 원본 상품 자체의 성분 분석은 반드시 수행한다.

## 상품명 정규화와 동일 상품 판단

### 문제

같은 상품이라도 쇼핑몰마다 상품명이 다르게 표시된다.

예:
- Nature Made Super B-Complex with Vitamin C, 140 Tablets
- 네이처메이드 슈퍼B WITH C
- NATURE MADE SUPER B

상품명 문자열만 비교하면 다른 용량, 다른 수량, 다른 제형 상품이 섞일 수 있다.

### 정규화 기준

상품명은 하나의 문자열로만 정규화하지 않고, 상품 정체성 필드로 나누어 정규화한다.

- brand: 브랜드명
- product_line: 제품 라인명
- form: 제형
- package_quantity_value: 포장 수량 숫자
- package_quantity_unit: 포장 수량 단위

예:
- brand: nature made
- product_line: super b complex with vitamin c
- form: tablet
- package_quantity_value: 140
- package_quantity_unit: tablets

### canonical_key

정규화된 상품 정체성 필드를 조합해 `canonical_key`를 만든다.

예:
```text
nature-made|super-b-complex-with-vitamin-c|tablet|140|tablets
```

`canonical_key`는 같은 상품 후보를 묶기 위한 1차 기준이다.

### 판매처별 원문 상품명

- `products.name`은 우리 서비스에서 보여줄 대표 상품명이다.
- `product_offers.source_product_name`은 각 쇼핑몰에서 수집한 원문 상품명이다.
- 판매처별 원문 상품명은 서로 다를 수 있다.

### 다른 쇼핑몰 가격 후보 처리

- 다른 쇼핑몰 URL을 이미 알고 있으면 해당 URL을 직접 수집한다.
- URL이 없으면 정규화된 브랜드, 제품명, 함량, 수량을 기반으로 검색한다.
- 검색 결과 상위 10개는 후보로만 사용한다.
- 검색 후보를 바로 최저가에 포함하지 않는다.

### LLM 검증 기준

검색 후보는 기준 상품과 아래 항목을 비교한다.

- 브랜드
- 제품 라인명
- 성분 또는 주요 기능
- 함량
- 제형
- 포장 수량
- 상품 이미지
- 옵션 텍스트

같은 상품으로 판단된 후보만 `product_offers`에 upsert한다.

### 최저가 기준

- 최저가는 검증 완료된 `product_offers`만 사용한다.
- 최저가는 배송비 포함 가격이다.
- 계산식은 `price + shipping_fee`이다.
- 검색 결과가 없거나 같은 상품 검증에 실패한 판매처는 최저가 비교에 포함하지 않는다.

## 상품 목적 태그

### 목적

상품 목적 태그는 상품 목록, 상품 상세, 대체상품 추천에서 사용자가 상품의 성격을 빠르게 이해하도록 돕기 위한 표시값이다.
태그는 질병 치료 효능을 보장하는 문구가 아니라 공공데이터 기능성 문구를 바탕으로 만든 관련 기능성 또는 섭취 목적이다.

### 생성 기준

- 공공데이터 `ingredient_public_sources.functionality`를 우선 사용한다.
- I2710, I-0040, I-0050의 기능성 문구를 규칙 또는 AI 보조로 표준 목적 태그에 매핑한다.
- 성분별 태그는 `ingredient_purpose_tags`에 저장한다.
- 상품별 태그는 상품에 포함된 성분의 태그를 집계해 `product_purpose_tags`에 저장한다.
- 상품별 태그 점수는 실제 저장된 성분 태그만 사용해 계산한다.
- `product_purpose_tags.ingredient_count`는 해당 목적 태그가 연결된 상품 성분 수이다.
- `product_purpose_tags.avg_confidence`는 연결된 `ingredient_purpose_tags.confidence` 평균이다.
- `product_purpose_tags.score`는 대표 태그 정렬용 점수이며, MVP에서는 `ingredient_count`와 `avg_confidence`만 반영한다.
- 상품 태그는 상품 성분 분석이 완료된 시점에 생성하거나 갱신한다.
- 상품명, 성분명, 성분 함량, 공공데이터 기능성 문구를 합친 정규화 텍스트로 상품 임베딩을 만들고 `product_embeddings`에 저장한다.
- 반입차단 공공데이터 `BLOCKED_IMPORT`만 있는 성분은 목적 태그 근거로 우선 사용하지 않는다.
- 공공데이터 기능성 근거가 없으면 상품 목적 태그를 만들지 않는다.

### 표시 기준

- 상품 목록에서는 대표 목적 태그를 최대 2개까지 표시한다.
- 대표 목적 태그는 `score DESC`, `ingredient_count DESC`, `avg_confidence DESC`, `purpose_tag ASC` 순서로 선택한다.
- 상품 상세에서는 목적 태그와 태그 근거 성분을 함께 표시할 수 있다.
- 문구는 `효능`보다 `관련 기능성`, `섭취 목적`, `함께 비교할 기준`처럼 단정적이지 않은 표현을 사용한다.
- 목적 태그가 없으면 해당 영역을 표시하지 않는다.
- 일반 상품의 함께 본 상품 추천은 사용자 조회 활동 집계를 기반으로 만든다.

## 상품 검색

### 검색 기준

- 상품명 기준 검색과 성분명 기준 검색을 동시에 수행한다.
- 성분명 검색은 `ingredient_aliases.normalized_alias`로 표준 성분을 찾고, `product_ingredients`에서 해당 성분이 포함된 상품을 찾는다.
- 상품명 검색 결과와 성분명 검색 결과는 `product_id` 기준으로 중복 제거한다.

### 검색 결과 표시

- 상품 행을 클릭하면 상품 상세 화면으로 이동한다.
- 상품 행에는 대표 목적 태그를 최대 2개까지 표시한다.
- 판매처 가격은 버튼이나 링크처럼 보이지 않게 표시한다.
- 가격은 배송비 포함 가격임을 사용자에게 보여준다.
- 반입차단 상품은 가격 대신 금지품목 경고를 표시한다.

## 상품 상세

### 초기 조회

상품 상세 화면은 초기 조회에서 아래 정보를 받는다.

- 상품 기본 정보
- 검증 완료된 판매처 가격
- 관심품목 담김 여부
- 상품 성분 목록
- 성분별 최종 분석 결과
- 내 정보 기준 주의 결과
- 내 그룹 목록
- 기본 그룹 기준 주의 결과
- 판매처별 댓글 요약
- 대표 목적 태그
- 대체상품 또는 추천 상품 목록

### 판매처별 댓글 요약

- 댓글 요약은 상품 전체가 아니라 `product_offers` 기준으로 관리한다.
- 배송, 포장, 판매자 응대는 쇼핑몰마다 경험이 다르므로 판매처별로 따로 보여준다.
- 같은 상품이라도 iHerb 댓글 요약과 Amazon 댓글 요약은 서로 다를 수 있다.
- 상세 화면에서는 각 판매처 가격 아래에 해당 판매처의 댓글 요약을 함께 보여준다.
- 댓글 요약은 긍정, 부정, 기타 비율을 표시한다.
- AI는 비슷한 댓글을 묶고, 댓글 또는 댓글 묶음을 긍정, 부정, 기타로 분류한다.
- 배송, 가격, 품질, 포장, 효과, 재구매 같은 관점별 요약을 만든다.
- “배송은 빠른데 품질은 별로”처럼 한 댓글에 여러 의견이 있으면 관점별 감성으로 나눠 해석한다.
- 리뷰 수집과 요약 데이터가 없으면 댓글 요약 영역은 표시하지 않는다.
- 쇼핑몰 댓글 수집은 각 쇼핑몰의 허용 방식과 약관을 확인한 뒤 진행한다.

### 성분 분석

- 상품 성분은 `product_ingredients`에 저장한다.
- 공공데이터 원천 근거는 `ingredient_public_sources`에 저장한다.
- 공공데이터별 주의사항은 상품 상세 진입 때마다 다시 비교하지 않는다.
- 분석 시점에 성분별 최종 분석 결과를 만들고 `product_ingredient_analyses`에 저장한다.

### MVP 위험 판정 목표

- MVP의 핵심은 상품 성분이 국내 반입차단 대상인지 확인하고, 사용자의 온보딩 정보 기준으로 주의가 필요한지 알려주는 것이다.
- 반입차단 성분은 개인 건강정보와 상관없이 최우선 위험으로 본다.
- 반입차단이 아닌 성분은 공공데이터의 기능성, 섭취량, 주의사항을 기준으로 사용자별 주의 여부를 판단한다.
- 사용자별 주의 판단은 구매를 차단하지 않고 노란색 주의 상태로 표시한다.
- 그룹별 주의 판단은 그룹원 중 한 명이라도 주의 대상이면 그룹원 주의로 표시한다.

### 상품 자체 위험 판정 순서

1. 상품에서 성분명과 함량을 추출한다.
2. 성분명을 `ingredient_aliases.normalized_alias`로 정규화해 표준 `ingredients`와 연결한다.
3. 표준 성분이 반입차단 공공데이터 `BLOCKED_IMPORT`에 있으면 해당 성분은 `BLOCKED`로 판정한다.
4. 하나 이상의 성분이 `BLOCKED`이면 상품 `products.status`도 `BLOCKED`로 판정한다.
5. 반입차단 성분이 없으면 I2710, I-0040, I-0050 공공데이터를 조회해 기능성, 섭취량, 주의사항을 합친다.
6. 합친 공공데이터 근거로 성분별 최종 분석 결과를 `product_ingredient_analyses`에 저장한다.
7. 성분별 최종 분석 결과 중 주의가 필요한 성분이 있으면 상품은 `CAUTION`, 없으면 `SAFE`로 판정한다.

### 공공데이터별 사용 기준

- `BLOCKED_IMPORT`: 국내 반입차단 여부 판단에 사용한다. 매칭되면 즉시 `BLOCKED`이다.
- `I2710`: 일반 건강기능식품 품목의 기능성, 일일섭취량, 섭취 시 주의사항을 사용한다.
- `I-0040`: 기능성 원료 인정현황의 기능성 내용, 1일 섭취량, 섭취 시 주의사항을 사용한다.
- `I-0050`: 개별인정형 원료의 기능성, 일일섭취량 상하한, 섭취 시 주의사항을 사용한다.
- 같은 성분에 여러 공공데이터 주의사항이 있으면 하나로 합쳐 성분별 최종 주의 근거를 만든다.
- 상품 상세에서는 공공 API를 매번 다시 호출하지 않고 저장된 공공데이터와 분석 결과를 사용한다.

### 온보딩 기반 사용자 위험 판정

사용자별 위험 판정은 `사용자 온보딩 최신 버전 + 상품 성분 분석 결과`를 비교해 수행한다.

1. 사용자의 최신 `user_onboardings.version`을 조회한다.
2. 상품의 `product_ingredients`와 `product_ingredient_analyses`를 조회한다.
3. 성분별로 `user_id + ingredient_id + onboarding_version` 캐시가 있는지 확인한다.
4. 캐시가 있으면 `user_ingredient_cautions`의 결과를 그대로 사용한다.
5. 캐시가 없으면 아래 온보딩 조건 매칭 규칙으로 1차 주의 여부를 계산한다.
6. 키워드만으로 판단이 애매하면 LLM 또는 pgvector 의미 비교로 보조 판단한다.
7. 판단 결과와 사용자에게 보여줄 문구를 `user_ingredient_cautions`에 upsert한다.
8. 주의 성분이 하나라도 있으면 사용자 기준 `CAUTION`, 없으면 `SAFE`로 응답한다.

### 온보딩 조건 매칭 규칙

- 온보딩 조건별 키워드는 코드에 직접 박지 않고 DB 또는 설정값으로 관리한다.
- 공공데이터 주의사항, 성분명, 성분 별칭, AI 요약 근거 텍스트를 대상으로 키워드 매칭한다.
- PostgreSQL에서는 `ILIKE` 또는 전문검색을 사용할 수 있다.
- MVP에서는 `ILIKE` 기반 키워드 매칭으로 시작하고, 누락이 많은 조건부터 의미 기반 비교를 추가한다.

- `is_pregnant = true`
  - 공공데이터 주의사항에 임산부, 임신, 태아, 비타민 A 과다, 레티놀, 호르몬 관련 주의가 있으면 경고한다.
  - 예: “임신 중 비타민 A 과다 섭취 주의”

- `is_breastfeeding = true`
  - 공공데이터 주의사항에 수유부, 모유, 영유아 영향, 카페인, 호르몬 관련 주의가 있으면 경고한다.

- `is_child = true`
  - 공공데이터 주의사항에 어린이, 영유아, 소아, 성장기, 섭취량 제한이 있으면 경고한다.

- `is_elderly = true`
  - 공공데이터 주의사항에 고령자, 노약자, 혈압, 혈당, 항응고제, 간/신장 기능 관련 주의가 있으면 경고한다.

- `has_hypertension = true`
  - 공공데이터 주의사항에 혈압, 심박, 심혈관, 카페인, 각성, 나트륨, 감초 관련 주의가 있으면 경고한다.
  - 예: 카페인 성분은 심박 증가나 혈압 상승 가능성이 있으면 경고한다.

- `has_hyperlipidemia = true`
  - 공공데이터 주의사항에 이상지질혈증, 고지혈증, 콜레스테롤, 중성지방, 간 기능, 지질대사 관련 주의가 있으면 경고한다.

- `has_diabetes = true`
  - 공공데이터 주의사항에 혈당, 인슐린, 당뇨, 당류, 탄수화물 대사 관련 주의가 있으면 경고한다.

- `is_caffeine_sensitive = true`
  - 성분명이 카페인 또는 카페인 별칭으로 매칭되면 경고한다.
  - 공공데이터 주의사항에 불면, 두근거림, 심박 증가, 각성 관련 문구가 있으면 경고한다.

- `additional_notes`
  - 사용자가 추가 입력한 건강 상태나 주의 성분은 키워드로 추출해 성분명, 별칭, 공공데이터 주의사항과 비교한다.
  - 예: 사용자가 “와파린 복용”을 입력했고 공공데이터 주의사항에 항응고제 관련 문구가 있으면 경고한다.

### 사용자 위험 문구 생성 기준

- 위험 문구는 사용자의 온보딩 조건, 성분명, 공공데이터 주의사항 근거를 함께 포함한다.
- 사용자에게 보여줄 문구는 LLM으로 자연스럽게 만들 수 있다.
- LLM은 없는 근거를 새로 만들지 않고 저장된 성분 분석 결과와 공공데이터 근거 안에서만 설명한다.
- 문구는 과장하지 않고 “주의가 필요합니다” 수준으로 표현한다.
- 확정적인 질병 진단이나 복용 금지 판단을 하지 않는다.
- 반입차단이 아닌 개인 주의는 구매 차단이 아니라 주의 안내로 표시한다.

예:
```text
고혈압 정보가 있어 카페인 성분에 주의가 필요합니다. 카페인은 심박 증가나 혈압 상승 가능성이 있어 섭취 전 확인이 필요합니다.
```

### 그룹 위험 판정

- 그룹 위험 판정은 그룹원 각각의 최신 온보딩 기준으로 사용자 위험 판정을 반복한 결과이다.
- 그룹원별로 `user_id + ingredient_id + onboarding_version` 캐시를 사용한다.
- 그룹원 중 한 명이라도 주의 결과가 있으면 그룹 기준 상태는 `GROUP_CAUTION`이다.
- 그룹 위험 결과에는 위험한 그룹원 이름, 성분명, 주의 사유를 함께 표시한다.
- 그룹원이 온보딩을 수정하면 새 온보딩 버전 기준으로 다시 계산한다.

### 상태 우선순위

상품과 화면 표시는 아래 우선순위를 따른다.

1. `BLOCKED`: 반입차단 성분이 포함된 상품. 구매 링크를 제공하지 않는다.
2. `GROUP_CAUTION`: 선택한 그룹원 중 주의 대상자가 있는 상품. 구매는 가능하지만 그룹원 주의를 표시한다.
3. `USER_RISK` 또는 `CAUTION`: 내 온보딩 기준 주의가 필요한 상품. 구매는 가능하지만 내 정보 기준 주의를 표시한다.
4. `SAFE`: 반입차단 성분도 없고 온보딩 기준 주의도 없는 상품.

### 구매 가능 여부

- `BLOCKED` 성분이 포함된 상품만 구매 링크를 제공하지 않는다.
- 내 정보 기준 주의나 그룹원 주의는 구매를 막지 않는다.
- 내 정보 기준 주의와 그룹원 주의는 노란색 주의 상태로 표시한다.

### 대체상품과 추천 상품

- 상품 상세 하단에는 상품 상태에 따라 추천 상품 영역을 표시한다.
- 현재 상품이 `BLOCKED`이면 `대체로 확인해볼 상품`을 표시한다.
- 대체상품은 현재 상품과 하나 이상의 목적 태그가 겹치고 pgvector 의미 유사도가 높은 상품 중 `BLOCKED`가 아닌 상품만 사용한다.
- 대체상품에는 겹치는 목적 태그, 의미 유사도 근거, 배송비 포함 최저가를 함께 표시할 수 있다.
- 현재 상품이 `SAFE`, `CAUTION`, `GROUP_CAUTION`이면 `이런 상품은 어떤가요`를 표시한다.
- 일반 상품 추천 영역은 `비슷한 목적으로 비교`와 `이 상품을 본 사람들이 함께 본 상품` 두 묶음으로 나눈다.
- `비슷한 목적으로 비교`는 현재 상품과 목적 태그가 겹치는 상품을 먼저 후보로 좁히고, `product_embeddings`의 pgvector 유사도 순으로 정렬한다.
- `이 상품을 본 사람들이 함께 본 상품`은 `product_view_events`를 집계한 `product_view_recommendations`를 사용한다.
- 함께 본 상품 추천은 사용자의 실제 조회 흐름 기반이므로 같은 목적 태그가 없어도 노출될 수 있다.
- 함께 본 상품 추천 문구는 행동 기반 추천으로만 표현한다.
- MVP에서 조회 로그가 충분하지 않으면 mock 또는 운영자가 만든 시드 집계를 사용할 수 있다.
- `비슷한 목적으로 비교`는 목적 태그 후보가 없으면 pgvector 유사도, 같은 성분, 같은 제품 라인, 비슷한 가격대 순으로 보조 후보를 찾을 수 있다.
- `이 상품을 본 사람들이 함께 본 상품`은 활동 집계 후보가 없으면 해당 섹션을 표시하지 않는다.
- 모든 추천 섹션에 후보가 없으면 추천 영역을 표시하지 않는다.
- 추천 문구는 치료 효과나 복용 효과를 단정하지 않는다.

### 목적 키워드와 pgvector 추천 랭킹

- 추천 시점에는 AI를 다시 호출하지 않고 저장된 `product_purpose_tags`와 `product_embeddings`를 사용한다.
- 목적 태그는 추천 이유를 설명하기 위한 1차 필터와 표시값으로 사용한다.
- pgvector 유사도는 같은 목적 태그 후보 안에서 더 비슷한 상품을 정렬하기 위해 사용한다.
- 최종 점수는 목적 태그 겹침 수, pgvector 유사도, 상품 상태, 가격 존재 여부를 조합해 계산한다.
- `SAFE` 상품은 `CAUTION` 상품보다 우선 노출할 수 있다.
- `BLOCKED` 상품은 대체상품, 비슷한 목적 추천, 함께 본 상품 추천 후보에서 제외한다.

### 사용자 활동 기반 추천 집계

- 상품 상세 진입 시 `product_view_events`에 조회 이벤트를 저장할 수 있다.
- 로그인 사용자는 `user_id`, 비로그인 사용자는 익명 세션 ID를 기준으로 집계한다.
- 같은 세션 또는 같은 사용자가 일정 시간 안에 함께 본 상품 쌍을 집계해 `product_view_recommendations`에 저장한다.
- 집계 점수는 함께 조회 횟수, 최신성, 중복 제거 정책을 반영한다.
- 반입차단 상품은 일반 상품의 함께 본 상품 추천 후보에서 제외한다.
- 조회 이벤트는 추천 품질 개선 용도이며, 건강 상태나 구매 여부를 추론하는 데 사용하지 않는다.

### 수집 정보

회원가입 또는 온보딩 수정에서 아래 정보를 받는다.

- 고혈압
- 고지혈증
- 당뇨
- 수유
- 어린이
- 고령자
- 카페인 민감
- 추가 정보

### 버전 관리

- 온보딩 정보는 `version`으로 관리한다.
- 사용자가 온보딩을 수정하면 새 버전으로 저장한다.
- 사용자별 성분 주의 캐시는 `user_id + ingredient_id + onboarding_version` 기준으로 관리한다.

## 사용자별 성분 주의 캐시

### 목적

상품 상세나 그룹 위험 조회 때마다 같은 사용자와 같은 성분을 반복 분석하지 않기 위해 사용한다.

### 캐시 키

```text
user_id + ingredient_id + onboarding_version
```

### 처리 흐름

- 캐시가 있으면 `caution_text`를 사용한다.
- 캐시가 없으면 사용자 온보딩 정보와 성분 분석 결과를 기준으로 주의 여부를 계산한다.
- 새로 계산한 결과는 `user_ingredient_cautions`에 upsert한다.

## 그룹

### 그룹 기능

- 사용자는 그룹을 만들 수 있다.
- 그룹은 초대코드로 참여할 수 있다.
- 그룹 참가 화면에서는 그룹명 또는 초대코드로 그룹을 검색한다.
- 그룹 참가 요청은 바로 가입이 아니라 가입 신청으로 저장한다.
- 가입 신청이 승인되면 그룹 멤버가 된다.
- 그룹 상세에서는 그룹원 목록과 온보딩 완료 여부를 볼 수 있다.
- 그룹장은 초대코드를 복사하고 그룹을 삭제할 수 있다.

## 인증과 마이페이지

### 로그인

- 사용자는 카카오 로그인으로 SafeBuy에 가입하거나 로그인한다.
- 처음 로그인한 사용자는 사용자 row를 생성한다.
- 이미 로그인한 적이 있는 사용자는 기존 사용자 정보를 사용한다.
- 회원가입 화면의 닉네임 input에는 카카오 프로필 닉네임을 기본값으로 넣는다.
- 사용자는 카카오에서 가져온 기본 닉네임을 가입 전에 수정할 수 있다.
- 사용자가 수정한 닉네임을 최종 서비스 닉네임으로 저장한다.

### JWT 인증

- 카카오 로그인 성공 후 SafeBuy 서비스용 access token과 refresh token을 발급한다.
- access token은 짧은 만료 시간을 사용한다.
- refresh token은 Redis 화이트리스트에 저장된 경우에만 유효하다.
- refresh token 재발급 시 기존 refresh token은 Redis에서 제거하고 새 refresh token을 저장한다.
- 로그아웃 시 현재 refresh token을 Redis 화이트리스트에서 제거한다.
- access token을 즉시 무효화해야 하면 남은 만료 시간 동안 Redis blocklist에 저장한다.
- 로그인 필요한 API는 `Authorization: Bearer {accessToken}`으로 인증한다.
- access token이 만료되면 refresh token으로 재발급한다.

### 마이페이지

- 마이페이지에는 닉네임, 닉네임 변경, 온보딩 수정, 로그아웃을 표시한다.
- 프로필 사진, 온보딩 완료 배지, 내 댓글 목록은 표시하지 않는다.

### 상품 상세의 그룹 위험

- 상품 상세 최초 진입 시 첫 번째 그룹을 기본 그룹으로 사용한다.
- 그룹을 변경하면 상품 전체를 다시 조회하지 않고 그룹 위험 결과만 재조회한다.
- 그룹 위험 결과도 `user_ingredient_cautions` 캐시를 우선 사용한다.

## 관심품목

### 전체 관심품목

- 사용자는 상품 상세에서 특정 판매처를 관심품목에 담을 수 있다.
- 전체 관심품목은 `user_favorite_offers` 기준으로 조회한다.
- 같은 상품이라도 iHerb 기준, Amazon 기준처럼 판매처별로 담을 수 있다.
- 전체 관심품목에서는 사용자/그룹 주의 상태를 계산하지 않는다.
- 반입차단 상품만 반입차단 의심으로 표시하고, 나머지는 안전으로 표시한다.

### 그룹별 관심품목

- 그룹별 관심품목은 `group_favorite_products` 기준으로 조회한다.
- 그룹별 관심품목은 상품 기준으로 저장한다.
- 판매처는 검증 완료된 판매처 중 배송비 포함 최저가를 보여준다.
- 그룹원 중 위험 대상자가 있으면 그룹원 주의로 표시한다.

### 판매처 바로가기와 구매 확인

- 관심품목의 판매처 바로가기를 누르면 외부 쇼핑몰로 이동한다.
- 판매처 바로가기 직전에 `purchase_confirmation_checks`에 구매 여부 확인 대기 항목을 저장한다.
- 사용자가 앱으로 바로 돌아오면 구매 여부를 확인할 수 있다.
- 사용자가 앱으로 바로 돌아오지 않아도 다음 홈 진입 때 구매 여부를 다시 확인할 수 있다.
- 사용자가 구매했다고 확인하면 `purchase_records`에 구매 기록을 저장한다.
- 구매 기록으로 저장된 확인 항목은 `PURCHASED`로 변경한다.
- 구매하지 않았다고 답한 확인 항목은 `DISMISSED`로 변경한다.
- 구매 기록에는 사용자, 상품, 판매처, 구매 당시 가격, 배송비, 구매 시각을 저장한다.
- 구매 당시 가격은 이후 판매처 가격이 갱신되어도 구매내역 기준으로 유지한다.

### 그룹 구매내역

- 그룹별 관심품목 화면에는 구매내역 탭을 제공한다.
- 구매내역에서는 그룹원이 누가 어떤 상품을 구매했는지 확인할 수 있다.
- 구매내역은 이름순 또는 가격순으로 정렬할 수 있다.
- 가격순 정렬은 구매 당시 상품 가격과 배송비를 합산한 금액 기준이다.
