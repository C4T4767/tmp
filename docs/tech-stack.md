# 기술스택

## 현재 레포에 적용된 스택

### 프론트엔드 프로토타입

- Next.js 16
- React 19
- TypeScript
- Tailwind CSS 4
- Radix UI
- lucide-react
- pnpm

현재 `apps/frontend`는 웹 프로토타입이다.
팀원에게 화면을 보여주고 기능 흐름을 검증하기 위한 용도이며, 실제 모바일 앱 구현과는 분리해서 본다.

## 실제 서비스 도입 예정 스택

### 모바일 앱

- React Native
- Android Studio
- Kotlin

React Native는 실제 앱 화면을 구현하는 주 스택으로 사용한다.
Android Studio는 Android 빌드, 에뮬레이터, 디버깅, 배포 준비에 사용한다.
Kotlin은 Android 네이티브 기능이 필요할 때 사용한다.

Kotlin이 필요한 대표 기능은 다음과 같다.

- 다른 쇼핑몰 앱 또는 브라우저에서 공유하기로 SafeBuy 앱에 URL 전달
- 앱 링크 또는 딥링크 처리
- Android 권한, 빌드 설정, 네이티브 모듈 연동

### 백엔드 API 서버

- Java
- Spring Boot
- Spring Security
- JPA 또는 MyBatis

Spring Boot는 서비스의 메인 API 서버로 사용한다.
인증, 사용자, 그룹, 관심품목, 구매확인, 상품 상세 조회, PostgreSQL 트랜잭션 처리는 Spring 서버가 담당한다.

Spring 서버가 담당하는 대표 기능은 다음과 같다.

- 카카오 로그인
- JWT 발급 및 검증
- 사용자 온보딩 저장
- 그룹 생성, 가입, 조회
- 상품 검색
- 상품 상세 조회
- 관심품목 저장 및 삭제
- 구매 여부 확인
- 구매 기록 저장
- Redis 랭킹 조회
- LLM 분석 서버 호출 및 결과 저장

### LLM 분석 서버

- Python
- FastAPI

FastAPI는 LLM, 크롤링, 성분 추출, 임베딩 생성처럼 Python 생태계가 유리한 작업을 담당한다.
메인 API 서버를 FastAPI로 대체하는 것이 아니라, Spring 서버 옆의 분석 전용 서버로 둔다.

FastAPI 서버가 담당하는 대표 기능은 다음과 같다.

- 쇼핑몰 상품 페이지 수집
- 상품명, 브랜드, 용량, 수량 정규화
- 성분명 추출
- 성분 alias 매칭 보조
- 공공데이터 기반 성분 분석 보조
- 동일 상품 후보 10개 LLM 비교
- 상품 이미지 비교 보조
- 판매처별 댓글 감성 분류와 관점별 요약
- 임베딩 생성

### 데이터베이스

- PostgreSQL
- pgvector

PostgreSQL은 서비스의 기준 데이터 저장소로 사용한다.
사용자, 그룹, 상품, 성분, 공공데이터, 관심품목, 구매기록, 분석결과를 저장한다.

pgvector는 임베딩 검색이 필요한 데이터에 사용한다.
다만 MVP에서 모든 위험 판단을 pgvector로 처리하지는 않는다.
고혈압, 임신, 고지혈증, 당뇨 같은 명확한 온보딩 값은 우선 키워드 규칙으로 판단하고, 자유 입력이나 의미 기반 매칭이 필요한 부분에 pgvector를 사용한다.

pgvector를 사용할 후보는 다음과 같다.

- 상품명 정규화 후보 검색
- 성분 alias 후보 검색
- 쇼핑몰별 같은 상품 후보 검색
- 사용자 추가 건강 메모와 성분 주의사항 의미 비교
- LLM 분석 결과의 유사 근거 검색
- 키워드로 놓친 주의 문구 후보 검색

### 캐시 및 랭킹

- Redis

Redis는 빠르게 변하는 데이터와 토큰 상태를 관리하는 데 사용한다.

Redis 사용처는 다음과 같다.

- 실시간 검색 TOP 10 랭킹
- refresh token 화이트리스트
- access token blocklist
- 사용자 세션 token_id 목록

실시간 검색 TOP 10은 Redis Sorted Set으로 관리한다.
Redis에는 `product_id`만 저장하고, 상품명과 이미지는 PostgreSQL에서 조회한다.

### 실시간 진행 상태

- SSE

상품 링크 분석은 여러 단계를 거치므로 SSE로 진행 상태를 전달한다.
클라이언트에는 기술 이벤트명이 아니라 사용자가 이해할 수 있는 문구로 보여준다.

예시는 다음과 같다.

- 쇼핑몰 확인 중
- 상품 정보 가져오는 중
- 성분 확인 중
- 안전성 확인 중
- 다른 판매처 가격 확인 중
- 분석 완료

## 스택 선택 기준

### React Native를 쓰는 이유

- 현재 화면 설계가 React 기반으로 진행되고 있다.
- Android 앱을 빠르게 만들 수 있다.
- 나중에 iOS가 필요해져도 화면 구조를 재사용하기 쉽다.
- Kotlin 네이티브 기능과 함께 사용할 수 있다.

### Spring Boot를 쓰는 이유

- 인증, 사용자, 그룹, 거래성 데이터 처리에 안정적이다.
- Java 기반 팀 개발과 운영에 익숙하다.
- PostgreSQL, Redis, JWT, Spring Security 조합이 일반적이다.
- 핵심 서비스 API를 한 곳에서 관리하기 좋다.

### FastAPI를 함께 쓰는 이유

- LLM, 임베딩, 크롤링, 텍스트 처리 라이브러리는 Python 쪽이 유리하다.
- 분석 작업을 Spring 서버에서 분리할 수 있다.
- 분석 서버 장애가 전체 서비스 API 장애로 번지는 것을 줄일 수 있다.

### pgvector를 쓰는 이유

- 상품명과 성분명은 쇼핑몰마다 표기가 다르다.
- 단순 문자열 검색만으로는 같은 상품과 같은 성분을 안정적으로 찾기 어렵다.
- PostgreSQL 안에서 벡터 검색을 같이 처리하면 별도 벡터 DB 없이 시작할 수 있다.

### Redis를 쓰는 이유

- 실시간 랭킹은 조회와 갱신이 자주 일어난다.
- JWT refresh token 화이트리스트와 로그아웃 처리를 빠르게 할 수 있다.
- DB에 저장할 필요가 낮고 만료 시간이 중요한 데이터를 관리하기 좋다.

## MVP 기준 역할 분리

### Spring Boot

- 클라이언트 API의 진입점
- 인증과 권한 처리
- PostgreSQL 트랜잭션 처리
- Redis 조회 및 갱신
- 분석 job 생성
- FastAPI 분석 요청
- 분석 결과 저장

### FastAPI

- 링크 수집
- 상품 정보 추출
- 성분 추출
- LLM 판단
- 댓글 감성 분류와 요약
- 임베딩 생성
- 동일 상품 후보 비교

### PostgreSQL

- 영속 데이터 저장
- 상품, 성분, 공공데이터, 사용자, 그룹, 구매기록 관리
- pgvector 기반 유사도 검색

### Redis

- 실시간 검색 랭킹
- 인증 토큰 상태
- 짧게 유지되는 캐시성 데이터

### React Native

- 실제 모바일 앱 화면
- 상품 링크 공유받기
- 홈, 검색, 상세, 관심품목, 그룹, 마이페이지 화면

## MVP에서 과하게 가지 않을 부분

- 모든 판단을 임베딩으로 처리하지 않는다.
- 온보딩의 명확한 건강 상태는 키워드 규칙으로 먼저 처리한다.
- pgvector는 문자열 검색이 어려운 후보 탐색과 의미 비교에 제한적으로 사용한다.
- FastAPI는 분석 전용으로 두고, 일반 서비스 API까지 모두 맡기지 않는다.
- Redis에는 영속 데이터 원본을 저장하지 않는다.
